# AI 应用开发方式评估与选型方案

> 编制日期：2026年3月
> 版本：v1.0

---

## 一、背景与目标

当前，大语言模型（LLM）技术已进入规模化落地阶段。企业在构建 AI 应用时，面临一个核心决策：**应该选择什么样的开发方式？**

本方案旨在从技术架构、开发效率、能力边界、成本效益等维度，系统性地评估三种 AI 应用开发路径，为后续项目选型提供决策依据。

### 领导关切的核心问题

1. 手动开发与 Dify 开发，架构上都是"提示词 + 知识库"，本质有何不同？
2. 对简单应用（智能客服、合同审核），两种方式优劣如何？
3. 更高级的 AI 应用应如何开发？
4. 知识库检索阈值难以确定的问题如何解决？

## 二、AI 应用开发的三条路径

### 2.1 完全手动开发（纯代码）

**代表技术栈**：Python + LangChain/LlamaIndex + 向量数据库 + 自建 API

**开发方式**：开发者从零搭建完整技术栈，包括：文档解析 → 文本分块 → 向量化 → 存储 → 检索 API → 提示词管理 → 对话管理 → 前端界面 → 部署运维。

**适用场景**：
- 需要深度定制底层算法和流程的场景
- 已有成熟技术团队且对 AI 技术栈有深入理解
- 对性能、延迟有极致要求的场景

**实际代价**：
- 一个基础的知识库问答系统，手动开发通常需要 **2-4 周**
- 需要解决的工程问题远超业务本身：embedding 模型选型、向量数据库选型与调优、分块策略、检索排序、提示词版本管理、对话历史管理、并发与性能、监控与日志等
- 后续每个功能迭代都涉及代码开发

### 2.2 低代码 AI 开发平台

**代表平台**：

| 平台 | 定位 | 核心优势 | 部署方式 |
|------|------|---------|---------|
| **Dify** | 开源 LLMOps 平台 | 全流程可视化、开源、生态活跃 | 云服务 + 私有化 |
| **Coze（扣子）** | 字节跳动低代码 AI 工厂 | 字节生态集成、上手极快 | 仅云服务 |
| **FastGPT** | 企业知识库问答系统 | RAG 能力深度优化、私有化 | 云服务 + 私有化 |
| **RAGFlow** | 深度文档理解 RAG 引擎 | 文档解析能力最强 | 开源私有化 |

**开发方式**：通过可视化界面拖拽编排工作流，配置提示词、知识库、工具调用，无需或少量编码即可构建 AI 应用。

**核心价值**：
- 开发周期从 **周级别缩短至天级别**（某电商案例：3 周 → 2 天）
- 内建完整 RAG 管道（分块、向量化、检索、重排序），无需从零搭建
- 企业级 LLMOps 能力：监控、日志、标注、提示词版本管理
- 团队聚焦业务逻辑，而非底层技术

### 2.3 手动开发 vs 平台开发：本质区别

> **领导的疑问**：两者不都是"提示词 + 知识库 → 大模型回答"吗？

**表面上看**，架构流程确实一致：用户输入 → 查询知识库 → 检索结果 + 提示词 → 大模型生成回答。

**但本质区别在于**：

| 环节 | 手动开发 | Dify 等平台 |
|------|---------|------------|
| **文档解析** | 需自行编写 PDF/Word/HTML 解析器 | 内置多格式解析，拖拽上传即用 |
| **分块策略** | 需自行实现分块算法、调优参数 | 提供多种分块模式，可视化配置 |
| **向量化** | 需集成 embedding 模型、管理向量库 | 一键配置，自动处理 |
| **检索策略** | 需自建向量检索 + 全文检索 + 融合排序 | 内置混合检索 + Rerank，参数可调 |
| **提示词管理** | 硬编码或自建版本管理 | 可视化编排、版本管理、A/B 测试 |
| **对话管理** | 需自建会话存储、上下文窗口管理 | 内建 Memory 管理 |
| **工具调用** | 需自行实现 Function Calling 逻辑 | 可视化配置工具、API 集成 |
| **监控运维** | 需自建日志、监控、告警体系 | 内置 LLMOps 监控面板 |
| **效果调优** | 修改代码 → 测试 → 部署 | 界面调参 → 即时测试 → 一键发布 |

**核心结论**：两者的底层原理相同，但平台把 **80% 的工程复杂度** 封装成了可配置项。就像"自己搭建 Web 服务器"与"使用 Nginx"的关系——底层都是 HTTP 协议，但工程效率天差地别。

## 三、Dify 的三种应用模式详解

Dify 提供三种核心应用模式，能力逐级递进：

### 3.1 聊天助手（Chatbot）— 入门级

**能做什么**：
- 基于提示词 + 知识库的一问一答
- 支持上下文对话（多轮对话记忆）
- 支持开场白、变量、引用标注

**做不到什么**：
- 不能调用外部工具或 API
- 不能执行多步骤逻辑
- 无法根据情况动态决策

**适用场景**：FAQ 问答、简单智能客服、知识库查询

**效果示例**：类似于当前市面上的基础智能客服——用户问"退货政策是什么？"，系统从知识库检索相关段落，结合提示词让大模型生成回答。

**与手动开发的对比**：
- 手动开发同样的 Chatbot，需要搭建完整的 RAG pipeline + 对话管理 + 前端界面
- Dify 上 **1-2 小时** 即可完成配置并上线

### 3.2 Agent（智能体）— 进阶级

**能做什么**：
- 在聊天助手基础上，增加 **工具调用** 能力
- LLM 自主推理判断，决定何时调用什么工具
- 支持 Function Calling 和 ReAct 两种推理模式
- 可集成外部 API、数据库查询、网络搜索等

**做不到什么**：
- 推理过程不透明（黑盒决策）
- 复杂流程控制不够精确
- 对于需要严格步骤的任务，可控性不足

**适用场景**：需要动态调用外部信息的客服（如查订单、查库存）、多工具协作的助手

**效果示例**：用户问"我的订单 2024001 到哪了？"，Agent 自主判断需要调用订单查询 API，获取物流信息后生成回答。

**与手动开发的对比**：
- 手动实现 Agent 需要构建 LLM 推理链、工具调度、错误处理等复杂逻辑
- Dify 上通过 **可视化配置工具列表** 即可实现，但深度定制仍需代码

### 3.3 工作流（Workflow / Chatflow）— 高阶级

**能做什么**：
- 基于节点的 **可视化流程编排**
- 支持条件分支、循环、并行处理
- 每个节点可以是：LLM 调用、知识库检索、工具调用、代码执行、HTTP 请求等
- 流程完全可控、可解释、可调试
- Chatflow 模式额外支持对话历史和多轮交互

**做不到什么**：
- 流程需要预先设计，灵活性不如 Agent 的自主推理
- 极端复杂的逻辑仍需代码节点辅助

**适用场景**：合同审核、多步骤文档处理、复杂业务流程自动化

**效果示例（合同审核）**：

```
用户上传合同
  → 文档解析节点（提取文本）
  → 知识库检索节点（匹配法规条款）
  → LLM 节点1（识别合同类型和关键条款）
  → 条件分支（按合同类型分流）
  → LLM 节点2（逐项审核条款合规性）
  → LLM 节点3（生成审核报告）
  → 输出节点（返回结构化审核结果）
```

**与手动开发的对比**：
- 手动实现同等复杂度的工作流，需要 **2-4 周** 开发 + 持续调试
- Dify 工作流可在 **2-3 天** 内完成编排和调试
- 关键优势：流程可视化 → 非技术人员也能理解和参与优化

### 3.4 三种模式对比总结

| 维度 | 聊天助手 | Agent | 工作流 |
|------|---------|-------|--------|
| **复杂度** | 低 | 中 | 高 |
| **开发时间** | 小时级 | 天级 | 天级 |
| **工具调用** | 不支持 | 自主调用 | 按编排调用 |
| **流程控制** | 无 | LLM 自主推理 | 显式节点编排 |
| **可解释性** | 低 | 中 | 高 |
| **可控性** | 高（但能力有限） | 中 | 高 |
| **适用场景** | 简单问答 | 多工具协作 | 复杂业务流程 |

**最佳实践**：在高级场景中，推荐 **混合架构**——用工作流驱动整体流程，在需要推理的节点嵌入 Agent，实现"流程可控 + 智能推理"的最佳组合。

## 四、知识库检索阈值问题的系统性解决方案

> **核心痛点**：对于用户的问题，向知识库进行查询时，检索结果的阈值难以确定——高阈值可能漏掉正确答案，低阈值引入过多噪声。

这不是一个单纯的"参数调优"问题，而是 RAG 系统设计中的系统性挑战。以下是多层次的解决方案：

### 4.1 为什么阈值难以确定？

在传统 RAG 系统中，检索结果的相似度分数受多个因素影响：

1. **文档类型差异**：不同类型文档（政策文件 vs 技术手册 vs 合同条款）的分数分布不同
2. **问题表述差异**：同一个意图的不同表达方式，匹配分数差距可能很大
3. **分块质量**：分块不合理导致语义碎片化，检索结果分散
4. **语义鸿沟**：用户用口语化表述，文档用专业术语，向量距离远

根据实证数据，内容检索准确率可达 83%，但最终 LLM 生成答案的精准度仅为 66%，两者间形成 17 个百分点的"可信度断层"。这揭示了单纯依赖检索阈值来保证答案质量是不够的。

### 4.2 数据层优化（源头治理）

| 策略 | 说明 | 效果 |
|------|------|------|
| **分块策略优化** | 针对不同文档类型选择合适的分块大小。技术文档推荐 400 词 + 80 词重叠；合同文档按条款自然分段 | 提升语义完整性，减少碎片化 |
| **父子分段模式** | Dify v0.15.0+ 支持：用小块（子块）精准匹配问题，用大块（父块）补充完整语境 | 同时解决"召回不准"和"上下文不足" |
| **元数据标注** | 为每个文档块添加分类、来源、适用场景等元数据，辅助检索过滤 | 精确缩小检索范围 |
| **QA 对生成** | 利用 LLM 为每个 chunk 生成它能回答的问题。查询时用"问题匹配问题"替代"问题匹配原文" | 显著提升语义匹配质量，缩小语义鸿沟 |

### 4.3 检索层优化（多策略融合）

| 策略 | 说明 | 效果 |
|------|------|------|
| **混合检索** | 同时使用向量检索（语义匹配）+ 全文检索（关键词匹配） | 互补两种方式的盲区，覆盖更广 |
| **多路召回** | 同时查询多个知识库，扩大召回范围 | 适用于知识分散在多个库的场景 |
| **Rerank 重排序** | 用专门的重排序模型（如 Cohere Rerank、bge-reranker、Jina Reranker）对初步检索结果进行语义重排 | 比原始向量相似度更准确的相关性判断 |
| **查询扩展** | 通过 LLM 将用户原始问题改写为多个语义相似的查询，多线并行检索后合并 | 长尾问题召回率提升 27% |

### 4.4 阈值策略建议

**核心原则**：不要依赖单一阈值，而是通过多层筛选来保证质量。

```
推荐配置流程：
1. 初步检索：TopK = 10-20（宽泛召回，确保不遗漏）
2. Rerank 重排序：对初步结果进行语义重排
3. 最终筛选：TopK = 3-5 + Score 阈值 0.3-0.5（在 Rerank 后生效）
```

**关键点**：
- 启用 Rerank 后，Score 阈值作用于重排序后的分数，而非原始检索分数
- 重排序模型的语义理解能力远强于简单的向量相似度
- 在 Rerank 后设置阈值的效果更稳定、更可靠
- Dify 的 Score 阈值默认关闭，建议开启并设置为 0.3-0.5（配合 Rerank 使用）

### 4.5 工程化优化

| 策略 | 说明 |
|------|------|
| **召回测试** | Dify 内置文本召回测试功能，可模拟用户问题并查看检索结果及分数，用于调优参数 |
| **RAGAS 评估框架** | 使用标准化评估工具量化 RAG 系统表现（忠实度、答案相关性、上下文精确度等） |
| **兜底机制** | 当检索结果均低于阈值时，不应强行回答，而是引导用户转人工或换一种方式提问 |
| **持续迭代** | 收集用户反馈 → 标注问答对 → 补充知识库 → 调优参数，形成闭环 |

### 4.6 前沿方向

| 技术 | 说明 | 成熟度 |
|------|------|--------|
| **GraphRAG** | 在向量检索基础上融入知识图谱，利用实体关系增强检索。适用于实体关系复杂的场景（如法规解读、合同条款关联分析） | 可试用 |
| **Agentic RAG** | 引入"反思"机制——Agent 评估检索结果是否足够，不够时自动改写查询、扩大搜索，直到获得满意结果 | 初步可用 |

### 4.7 小结：从"调阈值"到"系统性优化"

```
数据层：分块策略 + QA 对生成 + 父子分段 + 元数据标注
    ↓
检索层：混合检索 + 多路召回 + 查询扩展
    ↓
排序层：Rerank 重排序（核心环节）
    ↓
筛选层：TopK + Score 阈值（在 Rerank 后生效）
    ↓
兜底层：低分结果引导转人工
    ↓
迭代层：RAGAS 评估 + 用户反馈 + 持续优化
```

**结论**：阈值问题的根本解法不是"找到一个完美的阈值"，而是通过多层策略确保在到达阈值筛选环节时，候选结果的质量已经足够高。

### 5.2 关键技术趋势

#### MCP 协议（Model Context Protocol）

MCP 是 Anthropic 发布的开放标准协议，类比为"AI 的 USB-C 接口"，标准化了大语言模型与外部工具、数据源的交互方式。

- 将集成复杂度从"M×N"简化为"M+N"
- OpenAI、Google、阿里、腾讯等均已支持
- Gartner 预测：到 2026 年，70% 的企业将部署基于 MCP 的 AI Agent
- **采用 MCP 的开发效率提升 73%，集成成本降低 65%**

#### Agent 生态成熟

2026 年的 AI Agent 已从"能聊天的演示"向"可稳定交付的工程系统"演进：

- 四层架构逐渐清晰：Agent Loop（推理） + Runtime（执行环境） + MCP（连接） + Skills（方法论）
- 多 Agent 协作（A2A 协议）开始进入实用阶段
- 开发者在约 60% 的工作中使用 AI，但只能"完全委托"0-20% 的任务

#### RAG 技术演进

技术路线：传统 RAG → 混合检索 + Rerank → GraphRAG → Agentic RAG

| 指标 | 数值 |
|------|------|
| 优化后检索准确率 | 90%+ |
| 优化后生成准确率 | 85%+ |
| 某金融 RAG 系统 6 个月迭代后准确率 | 72% → 89% |
| 某客服系统人工介入率降低 | 100% → 不足 20% |

**建议**：智能客服属于标准场景，**强烈推荐使用 Dify 工作流模式开发**。将开发资源节省下来用于业务优化和知识库质量提升。

**推荐架构**：Dify Chatflow（对话工作流），包含以下核心节点：
```
用户输入
  → 意图识别（LLM 节点）
  → 条件分支
    ├── FAQ 类问题 → 知识库检索 → LLM 生成回答
    ├── 业务操作类 → 调用业务 API（工具节点）→ LLM 整理结果
```

### 6.2 合同审核

| 维度 | 手动开发 | Dify 开发 |
|------|---------|----------|
| 文档解析 | 需集成 PDF 解析库 | 内置多格式解析 |
| 条款拆分 | 自行实现 | 可通过分块策略配置 |
| 多模型协作 | 自行编排调用链 | 工作流节点编排 |
| 审核流程 | 代码实现逻辑分支 | 可视化条件分支 |
| 报告生成 | 自行开发模板系统 | LLM 节点 + 输出格式化 |
| 流程可视化 | 需额外开发 | 原生支持 |

**建议**：合同审核涉及多步骤流程，**Dify 工作流模式是最佳选择**。对于有特殊合规要求的条款匹配，可在工作流中嵌入代码节点做精确匹配。

**推荐架构**：Dify Workflow，包含以下核心节点：
```
合同文件上传
  → 文档解析节点（提取全文）
  → LLM 节点1（识别合同类型 + 提取关键条款）
  → 知识库检索节点（匹配法规/标准条款）
  → 条件分支（按合同类型分流）
    ├── 采购合同 → LLM 节点2a（审核采购特有条款）
    ├── 劳动合同 → LLM 节点2b（审核劳动法合规性）
    └── 其他类型 → LLM 节点2c（通用条款审核）
  → LLM 节点3（生成结构化审核报告）
  → 输出节点（JSON 格式报告 + 风险评级）
```

## 八、结论

### 回应领导关切

**Q1：手动开发和 Dify 开发架构一致，为什么还要用 Dify？**

底层原理确实一致，但 Dify 将 80% 的工程复杂度封装成了可配置项。就像用 Nginx 和自己写 HTTP 服务器——协议一样，但工程效率、稳定性、运维成本完全不同。**选择 Dify 是一个工程效率决策，而非技术路线决策。**

**Q2：简单应用（智能客服、合同审核）手动开发 vs Dify 优劣？**

对于这类标准场景，Dify 在开发速度（5-10倍）、维护成本、迭代效率上全面占优。手动开发的唯一优势是极端定制化能力，但在当前阶段，大多数需求 Dify 的工作流 + 代码节点已足以覆盖。

**Q3：更高级的 AI 应用怎么开发？**

采用"混合模式"：Dify 作为应用编排层 + 自定义 MCP Server 实现核心业务逻辑。这是 2026 年企业级 AI 开发的主流范式——通用能力用平台，核心逻辑用代码，通过 MCP 协议标准化连接。

**Q4：知识库检索阈值问题怎么解决？**

不依赖单一阈值。通过"混合检索 + Rerank 重排序 + 合理的 TopK/Score 配置 + QA 对生成 + 父子分段"等多层策略系统性解决。Dify 已内置这些能力，可直接配置使用。核心思路是：在到达阈值筛选环节时，确保候选结果的质量已经足够高。

---

### 一句话总结

> 在 2026 年初的时间节点，**以 Dify 为核心开发平台，采用"平台编排 + MCP 扩展"的混合架构**，是兼顾开发效率、技术可控性和未来扩展性的最优选择。
